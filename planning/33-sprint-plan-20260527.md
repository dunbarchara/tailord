## Day 6 — Gap Response Retrieval Quality

Two targeted fixes to ensure gap answers actually surface in LLM scoring context.

### Problem 1 — Gap prefix inflates embedding distance

`content` for `gap_response` claims includes display prefix `[Gap answer — {label}]: ...`. This is noise in embedding space — the cosine distance reflects the label, not the answer. Gap answers were competing poorly against resume bullets on exact-match queries.

- [x] `backend/app/services/experience_embedder.py` — added `_CLAIM_PREFIX_RE` regex and `_embedding_text(claim)` helper; strips `[Gap answer — ...]: ` prefix before calling `embed_text` for `gap_response` and `partial_response` source types; applied at all 3 embedding sites (`embed_experience_chunks` loop, `re_embed_chunk`)

### Problem 2 — Gap answers not guaranteed in top-K

Even with correct embeddings, if K resume bullets score higher cosine similarity, the gap answer is excluded from the LLM scoring context. A user who directly answers a requirement shouldn't lose to passive resume bulk.

### Design notes

- `Candidate Notes` (gap_response/additional_experience source types) is intentional scaffolding: once group-linking lands and claim normalisation at ingestion is in place, all claims regardless of source will be treated identically — routed into role buckets or `Additional Context` by group membership alone. `Candidate Notes` as a special section will be removed at that point. Source provenance will be preserved as inline `[gap_response]` attribution if the LLM signal remains useful.
- `Additional Context` is the permanent catch-all for ungrouped substantive claims (work_experience/project/other with no group_id/group_key). Mirrors the `Skills` / `Candidate Notes` header pattern.
- GitHub `detected_stack` claims are `claim_type="skill"` at ingestion — verified. They aggregate correctly under their parent role bucket via the inline `Skills:` label.

### Future: YOE derived from ExperienceGroup rows

`compute_profile_signals` currently derives total YOE from `work_experience[].duration` strings frozen in `ExperienceSource.source_data.extracted` at resume upload time. `"– Present"` roles resolve dynamically via `date.today()` so they tick up automatically, but new roles started after the last upload are invisible to the computation.

The correct long-term source of truth is `ExperienceGroup` rows (`group_type="role"`, `start_date`/`end_date` columns), which are kept current by claim ingestion. `end_date = None` should be treated as "Present". When groups are available, `compute_profile_signals` should compute the overlap-aware YOE merge from group date ranges instead of (or in addition to) the resume extract — the resume extract remains as a fallback for users with no groups.

This matters for long-term platform users who change jobs without re-uploading. Blocked on ensuring `ExperienceGroup.start_date`/`end_date` are reliably populated for resume-sourced role groups (currently set during `chunk_resume` in `experience_chunker.py`).

## Planning Note — GitHub Scan Destructiveness Scoping

Context: established that upload/connect is additive by default; destructive is opt-in (`destructive=False` default everywhere). GitHub rescan (light, README/metadata) is the one case that defaults to destructive for the scanned repos — it's a stateless snapshot refresh, so replacing existing claims is the correct semantic.

### Rule going forward

Destructiveness must be **scoped to the capture surface that produced the data**, not to the user's entire GitHub-derived corpus.

| Scan type | Destructive to | Notes |
|-----------|---------------|-------|
| Light rescan (README/metadata) | `source_type="github"` claims for that repo, produced by light scan | Stateless refresh — fine to replace |
| Deep scan (PR/commit-based, future) | Only claims produced by the deep scan surface for that repo | Stateful; rescan means "from last checkpoint forward." Full deep reset is explicit opt-in. |
| Continuous capture (future) | Never destructively overwritten by any manual scan | Append-only surface; separate `source_type` or `scan_surface` field distinguishes these claims |

### Data model implication

When deep scan and continuous capture exist, `source_type="github"` alone is insufficient to scope a destructive operation. Claims will need a `scan_surface` or `capture_mode` sub-field (e.g. `"light"`, `"deep"`, `"continuous"`) so that a rescan of the light surface does not touch continuous-capture claims for the same repo. This is a forward-looking requirement — implement alongside deep scan, not before.

### Current state

- Light rescan: `destructive=True` passed through `enrich_github_repos` → `chunk_github_repo`
- Fresh connect / add repos: `destructive=False` (additive)
- Resume replace: user-confirmed cascade choice, forwarded as `destructive` in `POST /experience/process`
- Resume fresh upload: `destructive=False` (additive)

## Day 23 — Scraping Security (SSRF) + Observability + Demo Dashboard Fix

### SSRF protection in the scraping pipeline

CodeQL flagged `py/full-ssrf` on `playwright_helper.py` — user-supplied URL passed directly to httpx/Playwright with no validation.

- [x] Added `_assert_public_url(url)` to `playwright_helper.py`: validates scheme (http/https only), blocks well-known internal hostnames (localhost, *.local, *.internal), resolves hostname via `socket.getaddrinfo`, rejects any resolved IP in 14 blocked networks (loopback, RFC 1918 private ranges, link-local 169.254.0.0/16 including cloud metadata endpoint, shared address space, multicast, reserved, IPv6 loopback/ULA/link-local)
- [x] Called at both entry points: `get_html_content` and `get_rendered_content`
- [x] Added `_validate_request_hook` (httpx request event hook): fires before every httpx request including redirect hops — closes the open-redirect bypass gap where a job board could redirect to 169.254.169.254 after the initial pre-flight check passes; registered via `event_hooks={"request": [_validate_request_hook]}`
- [x] Created `tests/unit/test_playwright_helper_ssrf.py` (28 tests): scheme checks, internal hostname checks, parameterised private IP matrix (13 IPs), IPv6 loopback, public IP pass-through, DNS failure, missing hostname, redirect hook blocks private IP, redirect hook allows public IP, redirect hook blocks loopback redirect

**Allowlist approach (rejected):** CodeQL suggested a fixed `_ALLOWED_JOB_HOSTS` set. Rejected — self-hosted job pages are a core use case; blocking by hostname would break the product. IP-range validation covers the real threat (internal infrastructure access) without restricting legitimate URLs.

**DNS rebinding note:** Pre-flight check does not fully prevent DNS rebinding (attacker could return public IP during check, private IP during connection). Full prevention requires a pinned-IP custom transport. Residual risk is negligible for a job-parsing workload; documented in code docstring rather than silently assumed away. Network-level NSG rules (block outbound 169.254.169.254 and RFC-1918 from scraping subnet) are the complementary layer to close this gap — deferred to infra sprint.

**Firecrawl / third-party scraper note:** If Firecrawl (or similar) is added as a fallback, their infrastructure makes the actual HTTP request — your server only calls their API. SSRF risk shifts to their infrastructure. `_assert_public_url` still runs before passing URLs to Firecrawl, which prevents using Tailord as a proxy to attack Firecrawl's own internal services (TOS/ethics concern, not a risk to Tailord's infra). Validation requirement: same check, same entry point.
