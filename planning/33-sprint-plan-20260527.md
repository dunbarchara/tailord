## Day 6 тАФ Gap Response Retrieval Quality

Two targeted fixes to ensure gap answers actually surface in LLM scoring context.

### Problem 1 тАФ Gap prefix inflates embedding distance

`content` for `gap_response` claims includes display prefix `[Gap answer тАФ {label}]: ...`. This is noise in embedding space тАФ the cosine distance reflects the label, not the answer. Gap answers were competing poorly against resume bullets on exact-match queries.

- [x] `backend/app/services/experience_embedder.py` тАФ added `_CLAIM_PREFIX_RE` regex and `_embedding_text(claim)` helper; strips `[Gap answer тАФ ...]: ` prefix before calling `embed_text` for `gap_response` and `partial_response` source types; applied at all 3 embedding sites (`embed_experience_chunks` loop, `re_embed_chunk`)

### Problem 2 тАФ Gap answers not guaranteed in top-K

Even with correct embeddings, if K resume bullets score higher cosine similarity, the gap answer is excluded from the LLM scoring context. A user who directly answers a requirement shouldn't lose to passive resume bulk.

- [x] `backend/app/services/chunk_matcher.py` тАФ added `_append_pinned_claims(top_k, job_chunk_id, user_id, db)` helper; claims linked to a specific `job_chunk_id` via `provenance_metadata["job_chunk_id"]` are always appended to the top-K set regardless of cosine rank
- [x] `_score_chunk_vector`: added `pinned_job_chunk_id: str | None = None` param; calls `_append_pinned_claims` after retrieval when set
- [x] `re_enrich_single_chunk`: passes `pinned_job_chunk_id=chunk_id` to `_score_chunk_vector` so rescoring always includes directly-linked gap claims

### TODO logged

- [x] Added TODO comment in `tailorings.py` near gap answer creation: normalize conversational gap responses into "claim-speak" before ingestion (separate task тАФ tracked in `planning/34-database-schema-review.md`)
