## Epic 1 — LLM Quality & Evaluation

**Vision:** Make every LLM call measurable, improvable, and validated by data rather than eyeballing. The eval pipeline is the foundation that makes all future prompt and architecture changes defensible.

### Feature 1.1 — Telemetry & Debug Snapshots
- **Task:** Level 2 — store `profile_snapshot` JSON on `Tailoring` at generation time (captures exact formatted profile sent to LLM, not current mutable experience)
- **Task:** Level 3 — `tailoring_debug_logs` table (batch payloads + responses + LLM call log), gated by `DEBUG_LOGGING_ENABLED` env flag + per-user `debug_logging` flag

### Feature 1.2 — Eval Pipeline
- **Task:** Build test set of (job URL, profile) pairs with human-labeled expected chunk scores and advocacy blurb quality ratings
- **Task:** Eval runner — re-run chunk matching on test set, compute agreement with human labels (exact match + off-by-one tolerance)
- **Task:** Diff view — side-by-side comparison of two runs (prompt A vs B, model X vs Y); highlight diverged chunks
- **Task:** CI integration — run eval on PR when `prompts/chunk_matching.py` changes; fail/warn if agreement drops below threshold

### Feature 1.3 — Scoring Architecture Improvements
*(Prerequisite: eval pipeline established as measurement baseline)*
- **Task:** Section pre-filtering — classify sections as evaluable/non-evaluable before enrichment; skip LLM calls for "Benefits", "About Us", "Compensation", "EEO" etc.
- **Task:** Candidate fact sheet indexing — single LLM call produces compact structured fact sheet from raw profile; all scoring calls use fact sheet instead of full profile
- **Task:** Targeted self-verification — for Gap scores on YOE requirements, named technologies, education: run focused binary verification call before persisting
- **Task:** Evidence extraction architecture — decompose chunk matching into Phase 1 (profile → explicit atomic evidence list) + Phase 2 (match requirement chunks against evidence list); validate against eval baseline before committing
- **Task:** Embeddings / pgvector — embed each experience bullet/skill at profile processing time; retrieve top-K semantically similar entries per requirement chunk at scoring time; `pgvector` extension on existing PostgreSQL
- **Task:** Prompt iteration — tighten `generate_tailoring` system prompt; add few-shot examples to profile extraction prompt

## Epic 2 — Richer Experience Data

**Vision:** The experience layer should be genuinely comprehensive — resume, GitHub (deep), manual input — so the platform has enough signal to advocate honestly for any job requirement.

### Feature 2.1 — GitHub Deep Crawl
- **Task:** GitHub Contents API integration — crawl high-signal files per repo: `README.md`, `CLAUDE.md`, `package.json`, `pyproject.toml`, `Dockerfile`, CI workflow files
- **Task:** Per-repo LLM enrichment pass — `readme_summary` + `detected_stack` (specific frameworks, not just language)
- **Task:** DB schema — `github_repo_details` JSON column on `experiences` (backward-compatible with existing `github_repos`)
- **Task:** Frontend — surface enriched repo data in ParsedProfile GitHub tab
- **Task:** (Future) GitHub contributor analysis — crawl contribution history to other repos (PRs merged, issues closed)

### Feature 2.2 — Experience Processing Bug Fixes
- **Task:** Fix race condition: GitHub added while resume is processing — GitHub endpoint should not touch `status` if currently `"processing"`; experience processor should update only its own columns to avoid clobbering concurrent writes

## Epic 3 — Candidate Empowerment Loop

**Vision:** Tailord should guide users to surface and articulate their experience, not just generate documents. Every gap becomes a targeted question. Every answer improves all future tailorings.

### Feature 3.1 — Post-Tailoring Gap Detection
- **Task:** `app/prompts/gap_analysis.py` + `app/services/gap_analyzer.py` — new LLM pass after tailoring generation identifying requirements not sourceable from profile
- **Task:** `ProfileGap` + `GapAnalysis` Pydantic schemas (job_requirement, question_for_candidate, context, source_searched)
- **Task:** Frontend — surface gap questions below generated tailoring; inline answer flow that stores responses as `user_input` source in Experience

### Feature 3.2 — Interactive Tailoring Format
*(Prerequisite: static tailoring surface mature, hosted fast model)*
- **Task:** New generation pass + output schema: `RequirementAnnotation` (requirement, category, candidate_evidence[], evidence_sources[], strength)
- **Task:** Frontend — annotated job description rendering with expandable/hoverable evidence panels per requirement; gaps shown honestly with follow-up prompt
- **Task:** `/u/{slug}/{tailoringSlug}` toggle — static vs interactive view

### Feature 3.3 — Public Profile Chat
*(Prerequisite: user-verified extracted_profile with high quality, hosted Haiku-class model, rate limiting)*
- **Task:** Chat interface in left pane of `/u/{slug}` — SSE streaming (infrastructure already exists)
- **Task:** Rate limiting on public profile chat endpoint
