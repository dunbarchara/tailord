**Excerpt from security conversation:** If backend ingress wasn't restricted, the standard defenses are:

- Request signing (HMAC) — instead of a static X-API-Key, each request is signed with a timestamp + nonce + payload hash. Backend verifies the signature and rejects anything older than ~30s. A leaked key alone isn't enough — you'd also need to forge the signature on each new request.
- Mutual TLS (mTLS) — backend requires a client certificate from the frontend container. Only the frontend (holding the cert) can establish a connection. Common in service meshes (Istio, Linkerd). Completely eliminates the header-spoofing surface.
- Short-lived service JWTs — frontend mints a JWT signed with a private key, backend verifies with the public key. The user claims (including user ID) are inside a tamper-proof envelope, not a plain header.
- VNet-level controls — private endpoints, subnet restrictions, so the backend is simply unreachable outside the VNet regardless of what credentials you hold.

For your Azure setup, VNet integration + private endpoints for the backend container is the natural upgrade path when that becomes a compliance requirement. It's noted as deferred in CLAUDE.md already.
