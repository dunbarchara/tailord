- [ ] DB schema — `github_repo_details` JSON column on `experiences` (additive; backward-compatible with existing `github_repos` metadata)
- [ ] Alembic migration
- [ ] Respect rate limits: cap at top 20 repos by `pushed_at`; skip archived and repos inactive > 2 years; log request count

**Deliverable:** When a user connects GitHub, the enrichment background task crawls file content and stores structured per-repo summaries + detected stacks.

## Day 8 — Post-Tailoring Gap Detection (Backend)

**Theme:** After a tailoring is generated, the platform knows both the job requirements and the candidate's profile. Use that to surface targeted follow-up questions — specific to this job, specific to this candidate.

- [ ] `app/prompts/gap_analysis.py` — system prompt + user template; task: identify requirements not sourceable from profile, generate specific follow-up questions (not generic)
- [ ] `ProfileGap` + `GapAnalysis` Pydantic schemas in `app/schemas/`
  ```python
  class ProfileGap(BaseModel):
      job_requirement: str
      question_for_candidate: str
      context: str           # why this matters for this specific role
      source_searched: str   # what source was checked

  class GapAnalysis(BaseModel):
      gaps: list[ProfileGap]
      sourced_claim_count: int
      unsourced_claim_count: int
  ```
- [ ] `app/services/gap_analyzer.py` — calls `llm_parse_with_retry`; runs after `_finalize_tailoring` as a chained background task
- [ ] Store gap analysis as `gap_analysis` JSON column on `Tailoring` (nullable); Alembic migration
- [ ] `GET /tailorings/{id}` response — include `gap_analysis` field
- [ ] `POST /tailorings/{id}/gap-answer` — accepts `{question_index, answer}`, appends answer to `experience.user_input_text`; triggers re-enrichment

**Deliverable:** Backend detects and stores gap analysis for every newly-generated tailoring.
