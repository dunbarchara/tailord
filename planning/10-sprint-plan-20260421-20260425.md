## Day 12 — Experience Chunking Model

**Goal:** Break `extracted_profile` JSON blob into first-class `ExperienceChunk` records.

### Data model
- [ ] New `ExperienceChunk` SQLAlchemy model:
  ```
  id, experience_id (FK), source (resume/github/user_input),
  claim_type (work_experience/skill/project/education/other),
  content (text of the claim), date_range (nullable),
  technologies (JSON array, nullable), position (int),
  embedding (nullable — pgvector column added later),
  created_at, updated_at
  ```
- [ ] Alembic migration: `create_experience_chunks`

### Chunking service
- [ ] `backend/app/services/experience_chunker.py`
  - `chunk_experience(experience_id)` — reads `extracted_profile`, produces `ExperienceChunk` records
  - Deterministic split: one chunk per bullet point / work item / skill cluster
  - Idempotent: delete-and-replace on re-process
  - Triggered from `experience_processor.py` after profile extraction completes
- [ ] Unit tests: `tests/services/test_experience_chunker.py`

### API surface (read-only for now)
- [ ] `GET /experience/chunks` — returns chunked experience for the current user
- [ ] Frontend: no UI changes needed yet; chunks are internal

### Acceptance
- Re-processing experience populates `ExperienceChunk` records
- Existing `extracted_profile` JSON remains unchanged (chunks are additive)
- `make check-backend` passes

## Day 13 — pgvector Infrastructure + Embedding Pipeline

**Goal:** Add vector storage and embed experience chunks and job chunks on write.

### Infrastructure
- [ ] Add `pgvector` extension to PostgreSQL (migration + Docker Compose update)
- [ ] Add `embedding` column (`Vector(1536)`) to `ExperienceChunk` and `JobChunk`
- [ ] Alembic migration: `add_embedding_columns`
- [ ] `backend/app/clients/embedding_client.py`
  - Wraps OpenAI embedding API (text-embedding-3-small or configurable)
  - `embed_text(text: str) -> list[float]`
  - Same `LLM_BASE_URL` pattern for provider-agnosticism

### Embedding pipeline
- [ ] `ExperienceChunk`: embed on create/update (background task from `experience_chunker.py`)
- [ ] `JobChunk`: embed on create (background task from job parsing)
- [ ] Re-embed trigger: when `experience_processor.py` re-runs, re-embed affected chunks
- [ ] Embedding model name stored in `ExperienceChunk.embedding_model` (nullable string) — so we know when to re-embed after model changes

### Acceptance
- New experience chunks have non-null embeddings after processing
- New job chunks have non-null embeddings after job parsing
- No embedding failures block the main processing path (background task, non-fatal)

## Day 14 — Vector-Assisted Matching

**Goal:** Replace LLM chunk scoring with cosine similarity over embeddings.

### Matching service rewrite
- [ ] `backend/app/services/chunk_matcher.py` — new `enrich_job_chunks_vector()` path:
  - For each `JobChunk`, find top-k `ExperienceChunk` by cosine similarity
  - Thresholds (tunable): similarity ≥ 0.85 → score 2, ≥ 0.70 → score 1, else 0
  - Store `match_rationale` as "vector similarity: {score:.3f}" for traceability
  - LLM still used for `advocacy_blurb` generation (score ≥ 1 chunks only)
- [ ] Feature flag: `MATCHING_MODE=vector|llm` env var (default `llm` during transition)
- [ ] `re_enrich_single_chunk` updated to use vector path when flag is set

### Eval regression check
- [ ] Run `make eval` against vector-matched results
- [ ] Compare recall / precision against LLM baseline fixtures
- [ ] Document delta in `tests/eval/RESULTS.md`

### Acceptance
- `MATCHING_MODE=vector` produces scores that pass the eval harness baseline
- `MATCHING_MODE=llm` still works (no regression)
- Admin can toggle flag without code deploy
