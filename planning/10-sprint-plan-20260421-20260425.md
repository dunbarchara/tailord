## Day 14 ŌĆö Vector-Assisted Matching

**Goal:** Use cosine similarity to pre-select relevant ExperienceChunks per JobChunk, reducing
the LLM's input context from the full profile to a focused, relevant subset. LLM retains
STRONG/PARTIAL/GAP scoring responsibility ŌĆö vector similarity is a context reducer, not a scorer.

### Why not pure cosine thresholds?
Cosine similarity is a retrieval mechanism. It finds semantically proximate chunks but cannot
reason about temporal constraints ("5+ years"), contextual relevance (TypeScript in a core role
vs. mentioned in passing), or the nuanced difference between STRONG and PARTIAL evidence.
Replacing LLM scoring with fixed thresholds (e.g. Ōēź 0.85 = STRONG) would degrade match fidelity.
The correct use: surface the top-K most relevant chunks so the LLM reasons against focused evidence
rather than an entire profile.

### Context-attached retrieval (preserving LLM reasoning quality)
Embeddings are computed on **claim content only** ŌĆö the raw bullet or skill text, without position
metadata. This keeps cosine similarity semantically pure (matching "what was done", not "where").

Context is attached at retrieval time when building the LLM prompt. Each top-K chunk is annotated
with its structured metadata (`group_key`, `date_range`, `source_ref`, `technologies`) and chunks
from the same position are grouped before being passed to the LLM:

```
Requirement: "5+ years of TypeScript experience"

Relevant experience:
  Software Engineer @ ACME  (2020ŌĆō2025)
  ŌĆó Developed and maintained backend microservices in TypeScript and Go
  ŌĆó Owned TypeScript migration of the core payments module

  GitHub: tailord  (Python, FastAPI, TypeScript)
  ŌĆó Full-stack web application with Next.js frontend and FastAPI backend
```

This preserves the LLM's ability to name the company, role, and years in advocacy statements
without embedding that context into the chunk (which would distort similarity matching).

### Matching service update
- [ ] `backend/app/services/chunk_matcher.py` ŌĆö new `enrich_job_chunks_vector()` path:
  - For each `JobChunk`, run cosine similarity against all `ExperienceChunk` rows for the user
  - Retrieve top-K (default K=8, tunable) by similarity score
  - Build grouped context prompt: annotate each chunk with `group_key`/`date_range`/`source_ref`; group by position before passing to LLM
  - LLM assigns `match_score` (0/1/2) + `advocacy_blurb` in a single call using updated prompt (see below)
  - Store `match_rationale` with "vector pre-selection: top-{K} chunks used" for traceability
- [ ] **Update LLM scoring prompt** ŌĆö `chunk_matcher.py` scoring call changes input format from
  `(job_requirement, full_profile_string)` to `(job_requirement, grouped_context_chunks[])`;
  output schema unchanged (STRONG/PARTIAL/GAP + advocacy_blurb); update prompt in
  `backend/app/prompts/` accordingly
- [ ] Feature flag: `MATCHING_MODE=vector|llm` env var (default `llm` during transition)
  - `llm` mode: existing path (full formatted profile passed to LLM scorer)
  - `vector` mode: cosine pre-selection ŌåÆ context-attached grouped chunks ŌåÆ LLM scores
- [ ] `re_enrich_single_chunk` updated to use vector path when flag is set

### Eval regression check

- [ ] Compare recall / precision against LLM-on-full-profile baseline fixtures
- [ ] Confirm: STRONG/PARTIAL/GAP assignment quality is preserved or improved
- [ ] Document delta in `tests/eval/RESULTS.md`
